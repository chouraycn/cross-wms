OpenClaw vs CrossWMS（CDF KnowClow）

完整度与未来方向对比分析 · 生成于 2026-07-11 · 基于两仓库实际代码与文档

一句话结论

OpenClaw 是"广度碾压、平台/厂商/通道无关"的个人 AI 编排运行时（≈175 万行，25+ 通道 / 45+ 模型商 / 145 扩展，自托管 Gateway + 多端原生 App），但自评仍处 Beta（完整度 70%）、"still early"。
CrossWMS 是"纵深专精、单桌面、领域垂直"的跨境支付 WMS AI 助手（≈11 万前端行 + 894 服务端文件，v1.7.81），在流式体验、WKWebView/macOS 适配、领域路由上打磨更深，但缺少多通道 / 多平台 / 插件生态。
二者不是替代关系：CrossWMS 的护城河是"跨境支付领域深度 + 桌面流式体验"，应向 OpenClaw 借的是插件化架构、配置 doctor 迁移、安全沙箱、长会话压缩，而不是去复制它的全平台野心。

五、给团队的 actionable 建议

① 不要去复制 OpenClaw 的全平台野心

  1. CrossWMS 的护城河是跨境支付领域深度 + 桌面流式体验，不是"又一个通用助手"。
  2. 资源应投向领域数据闭环（WMS/双海关/价格口径）与流式打磨，而非 25+ 通道。

② 向 OpenClaw 借"架构化能力"，而非"功能清单"

  1. 插件 SDK + 市场思维：把 builtin/plugin/MCP 升级为带正式 SDK 的扩展体系，核心瘦身（呼应 OC "Core stays lean"）。
  2. 配置 doctor 迁移：破坏性配置变更配套 `doctor --fix`（自动备份+改写），降低升级摩擦。
  3. 长会话压缩/记忆：WMS 长对话易超上下文，引入 OC 式 compaction + 多后端记忆。
  4. 安全沙箱：high-risk 工具调用加 Docker/隔离执行，对齐 OC 的 sandbox.mode。

③ 补一份"能力成熟度评分卡"

  1. OpenClaw 用 scorecard（50 surfaces/281 areas）诚实标注 Beta/Alpha；CrossWMS 也可建立类似清单，避免"感觉完成"与"实际完成"的偏差。
  2. 对"生成文件""新建对话"等近期 UI 改动，建议纳入回归基线（你当前的 QA 严格度正好用得上）。
