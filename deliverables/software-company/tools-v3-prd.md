## 1. 背景与目标

### 1.1 现状

CrossWMS 当前工具系统（v2.x）包含三个 Web 工具——`web_search`、`web_fetch`、`web_api_call`，通过 `toolExecutor.ts` 统一调度执行。AI 助手的深度思考（`reasoning_content`）阶段是**固定、不可中断**的：思考过程中无法调用任何工具，只有当思考阶段完全结束后，才能进入工具执行阶段。

这带来了几个明显痛点：

- **思考与行动割裂**：AI 在思考阶段可能"想到"需要查库存或搜文档，但必须等思考流全部完成才能执行工具，导致结果无法影响后续思考分支。
- **工具扩展困难**：新增一个工具需要修改核心执行链（`chainExecutor.ts` / `toolExecutor.ts`），无法由第三方或运营人员动态注入。
- **浏览器能力缺失**：跨境仓库日常运营中大量操作涉及网页版后台（登录、查单、对账），当前 AI 无法直接操控浏览器，需用户手动截图或复制粘贴。
- **HTTP 集成薄弱**：`web_api_call` 仅支持基础 GET/POST，无模板管理、无鉴权配置、无响应提取，无法对接飞书/Slack/ERP 等内外部系统。

### 1.2 v3.0 核心目标

本次升级围绕 **"让 AI 在思考时也能行动、让工具生态可插拔、让浏览器和 HTTP 成为一等公民"** 三个核心理念，实现工具系统的代际跨越：

| 目标 | 描述 |
|------|------|
| **思考即行动** | 深度思考过程中支持 Plugin 自动注入工具结果，打破思考与执行的串行壁垒 |
| **开放生态** | 建立 Plugin Registry，支持插件生命周期管理（注册/激活/禁用/卸载），允许动态扩展工具 |
| **浏览器原生** | 提供 CDP + Playwright 双引擎浏览器工具，支持截图 + 快照 + Profile + 表单填充 |
| **HTTP 全面升级** | 扩展 web_api_call 为完整 HTTP 工具簇，支持模板、鉴权、响应提取、域名白名单动态管理 |
| **安全可控** | 域名白名单统一管理、SSRF 防护、权限分级（低权 CDP / 高权 Playwright / 无头/有头切换） |

## 2. 用户故事

### 方向1：Plugin System 动态注入

**US1-1 — 运营人员在思考阶段实时查库存**

> 作为仓库运营人员，当我向 AI 询问"请帮我分析华东仓最近一周的库存周转率"时，我希望 AI 在思考分析的过程中就能自动调用库存查询插件，将库存数据注入到思考流中，这样 AI 的分析结论可以直接基于真实数据，而不是靠推测。

**Acceptance Criteria:**
- 当 AI 进入深度思考模式时，已激活的 Plugin 在 stream 中被自动检测并调用
- Plugin 返回的结果实时回填到 `reasoning_content` stream 的当前上下文
- 用户可见思考流中注入了一段标记为 `[Plugin: inventory-query]` 的中间结果

**US1-2 — 开发者动态注册自研工具**

> 作为开发者，我希望能够将一个 Node.js 模块或 WASM 模块打包为 Plugin，通过 Plugin Registry API 注册到系统中，无需修改 `chainExecutor.ts` 核心代码。注册后该 Plugin 即可在 AI 思考阶段和工具执行阶段被调用。

**Acceptance Criteria:**
- 提供 `PluginRegistry.register(manifest)` API，支持运行时注册
- `manifest` 包含 name、version、hooks（`onReasoning` / `onToolCall` / `onResult`）
- Plugin 的注册、激活、禁用、卸载均通过生命周期 hook 管理
- 提供一个 `/plugins` 前端面板用于可视化 Plugin 管理

**US1-3 — Plugin 市场与发现**

> 作为运营管理员，我希望能在系统内置的 Plugin 市场中浏览、搜索和安装团队共享的插件（如"飞书消息通知插件""速卖通订单同步 Plugin"），无需开发者介入即可扩展系统能力。

**Acceptance Criteria:**
- Plugin 市场面板支持按分类/标签浏览
- 支持一键安装/卸载
- 安装后自动激活，并在 AI 下一次会话中生效

### 方向2：Browser 工具

**US2-1 — AI 自动登录仓库后台查单**

> 作为仓库操作员，当我向 AI 说"帮我去速卖通后台查一下订单 #8843921 的物流状态"时，AI 应该能自动打开浏览器（默认无头模式），使用已保存的速卖通 Profile 自动登录，导航到订单详情页，截图并提取关键信息返回给我。

**Acceptance Criteria:**
- 提供 `browser_navigate` / `browser_screenshot` / `browser_snapshot` 三个核心工具
- AI 自动选择 Playwright 引擎（已保存 Profile 场景）
- 截图 + 快照中交互元素带 ref ID，支持后续操作（click/type）
- 操作完成后以无头模式默认返回结果

**US2-2 — 用户确认后切换有头模式手动接管**

> 作为仓库管理员，当 AI 在浏览器中执行表单填写（如报关单）时，我希望 AI 填写完成后不要直接提交，而是切换到有头可视化模式让我确认，确认无误后我再点击"提交"或让 AI 继续执行。

**Acceptance Criteria:**
- AI 默认使用无头模式执行浏览器操作
- 在敏感操作（表单提交、支付）执行前，系统弹出"切换到有头模式确认？"对话框
- 用户确认后，Playwright 切换为有头模式，展示当前浏览器状态
- 用户可手动操作或点击"继续"按钮让 AI 执行提交

**US2-3 — 跨站点运营多 Profile 管理**

> 作为跨境运营主管，我的团队需要管理多个平台的账号（速卖通、Shopee、Lazada、1688），我希望 AI 可以根据任务上下文自动选择正确的浏览器 Profile，免去每次手动登录的麻烦。

**Acceptance Criteria:**
- Profile 管理面板支持创建/编辑/删除 Profile
- 每个 Profile 保存独立 localStorage + Cookie + 截图标识
- AI 在工具调用时可指定 `profile: "shopee-sg"`
- CDP 引擎模式下，Profile 对应一个已打开的 Chrome 用户数据目录

### 方向3：HTTP 工具整合

**US3-1 — 一键发送飞书通知**

> 作为运营人员，当仓库出现异常（如库存预警）时，我希望 AI 能通过预配置的飞书 Webhook 模板，自动发送通知到指定飞书群，无需每次手动填写 Webhook URL 和消息格式。

**Acceptance Criteria:**
- 提供 `api_template` 系统，预定义飞书/Slack/企业微信等常用模板
- 模板包含 URL、Method、Headers、Body Schema、Auth 配置
- AI 调用时只需指定 `template: "feishu-alert"` + 参数
- 支持模板参数替换（如 `{{message}}` → 实际内容）

**US3-2 — 监听外部 Webhook 并触发工作流**

> 作为仓库管理员，我希望当 ERP 系统通过 Webhook 推送"新订单"事件时，AI 能自动解析订单数据，查询库存，并预填出库单。我不需要手动转发数据给 AI。

**Acceptance Criteria:**
- `webhook_listen` 工具支持注册 HTTP 端点并接收外部 POST
- 接收到的 payload 自动传入 AI 上下文
- 支持关联 Action：定义"当收到 Webhook X 时，自动执行工具 Y"
- Webhook 监听列表可在面板中查看和管理

**US3-3 — API 响应精准提取**

> 作为运营数据分析师，当我让 AI 调用第三方物流 API 查询包裹轨迹时，返回的 JSON 通常非常庞大。我希望 AI 能通过 JSONPath 自动提取关键字段（如 `$.data.tracks[*].status`），只将精简后的信息呈现在我面前。

**Acceptance Criteria:**
- `web_api_call` 支持 `extract` 参数，接受 JSONPath 或 CSS Selector
- 提取结果以结构化形式展示（表格/列表）
- 支持多条 JSONPath 规则，每条规则可命名别名
- CSS Selector 模式用于 HTML 响应提取

## 3. 需求池与优先级

### 3.1 方向1 — Plugin System

| ID | 需求 | 优先级 | 说明 |
|----|------|--------|------|
| P1-01 | Plugin Registry 核心框架（注册/激活/禁用/卸载生命周期） | **P0** | 基础设施，所有 Plugin 功能依赖此框架 |
| P1-02 | 深度思考期间 Plugin 自动调用与结果回填 | **P0** | 核心增量价值，直接解决"思考与行动割裂" |
| P1-03 | chainExecutor 重构：支持 Plugin Hook 注入点 | **P0** | `onReasoning` / `onToolCall` / `onResult` 三个 hook |
| P1-04 | Plugin 安全沙箱（权限声明、资源限制） | **P1** | 防止恶意 Plugin 访问文件系统或网络 |
| P1-05 | `/plugins` 可视化面板（Plugin 列表、启用/禁用开关） | **P1** | 基础管理 UI |
| P1-06 | Plugin 市场的发现与安装 | **P2** | 生态建设，可在 v3.1 实现 |
| P1-07 | WASM Plugin 运行时支持 | **P2** | 高性能场景，可在 v3.2 实现 |

### 3.2 方向2 — Browser 工具

| ID | 需求 | 优先级 | 说明 |
|----|------|--------|------|
| P2-01 | Playwright 引擎集成（独立浏览器进程、无头/有头切换） | **P0** | 浏览器工具的底层引擎 |
| P2-02 | `browser_navigate` + `browser_screenshot` + `browser_snapshot` | **P0** | 最小可用浏览器工具集 |
| P2-03 | AI 快照 ref 引用（DOM 快照 + 可交互元素 ID 列表） | **P0** | Click/Type 操作依赖 ref 定位 |
| P2-04 | `browser_click` + `browser_type` 交互工具 | **P1** | 基于快照 ref 的交互操作 |
| P2-05 | 域名白名单（浏览器工具独立 + 与 HTTP 工具共享） | **P1** | SSRF 防护 |
| P2-06 | Profile 管理（保存/恢复 localStorage + Cookie） | **P1** | 多平台账号管理 |
| P2-07 | CDP 引擎（连接已有 Chrome） | **P1** | 低权限场景，复用已登录的浏览器 |
| P2-08 | 表单自动填充（字段描述→DOM 匹配→填入） | **P2** | 智能表单处理 |
| P2-09 | 有头模式可视化确认流程（敏感操作前） | **P2** | 需要前端配合弹窗交互 |

### 3.3 方向3 — HTTP 工具整合

| ID | 需求 | 优先级 | 说明 |
|----|------|--------|------|
| P3-01 | 扩展 `web_api_call` 支持 PUT/PATCH/DELETE | **P0** | 补齐 REST 动词，当前仅 GET/POST |
| P3-02 | API 请求模板系统（预定义模板 + 参数替换） | **P0** | 飞书/Slack/企业微信模板为 P0 |
| P3-03 | 域名白名单从 11 个扩展并支持动态增删 | **P0** | 安全基础 + 运行时灵活性 |
| P3-04 | Auth 配置管理（Bearer Token / API Key / OAuth2） | **P1** | 第三方 API 对接必备 |
| P3-05 | Response 提取（JSONPath / CSS Selector） | **P1** | 精准数据提取 |
| P3-06 | `webhook_listen`（监听外部 Webhook） | **P1** | 事件驱动工作流 |
| P3-07 | `webhook_send`（主动发送 Webhook） | **P1** | 与模板系统配合使用 |
| P3-08 | 请求历史与重放 | **P2** | 调试与审计 |
| P3-09 | API 组合编排（链式调用） | **P2** | 高级工作流，可在 v3.2 实现 |

## 5. 技术约束与待确认问题

### 5.1 已知技术约束

| # | 约束 | 影响方向 | 说明 |
|---|------|----------|------|
| C1 | pywebview 不支持多窗口 | 浏览器工具 | 有头模式下 Playwright 浏览器需独立进程，不能在 webview 内嵌；需确认窗口管理器如何协调前景/后台 |
| C2 | Node.js 后端非主进程渲染 | Plugin 沙箱 | Plugin 运行在 Node.js 后端进程，无法直接访问 DOM，需通过 IPC 与前端通信 |
| C3 | CDP 依赖已有 Chrome 实例 | 浏览器工具 | 用户必须提前启动 Chrome 并开启 `--remote-debugging-port`，否则 CDP 模式不可用 |
| C4 | 深度思考流由 LLM 服务端控制 | Plugin System | reasoning_content 的注入点是客户端行为，需确认 LLM API 是否支持 `interrupted` 模式或仅能在客户端拼接 |
| C5 | 域名白名单当前硬编码在 `toolExecutor.ts` | HTTP 工具 | 动态增删需迁移到配置文件或数据库，并确保热加载不中断工具执行 |
| C6 | Profile 的 Cookie 存储安全性 | 浏览器工具 | 需加密存储，避免明文 Cookie 泄露；考虑用系统 Keychain（macOS）/DPAPI（Windows） |
| C7 | 现有 3 个 Web 工具的实现复杂度 | 全部 | `chainExecutor.ts` / `toolExecutor.ts` 重构需向后兼容，v2.x 工具不能破坏 |

### 5.2 待确认问题

| # | 问题 | 相关方 | 紧迫度 |
|---|------|--------|--------|
| Q1 | **深度思考流的中断与恢复机制**：LLM 服务端是否支持在 reasoning 阶段接收外部信号（tool_result）并调整后续思考？还是必须在客户端侧用特殊标记拼接？ | 后端/LLM 集成 | 🔴 高 |

### 5.3 安全风险矩阵

| 风险 | 严重度 | 缓解措施 |
|------|--------|----------|
| Plugin 恶意代码 | 🔴 高 | 沙箱隔离、权限声明、代码签名、人工审核 |
| SSRF 攻击 | 🔴 高 | 域名白名单强制校验、内网 IP 拒绝、URL 解析规范化 |

| Webhook 端口暴露 | 🟡 中 | 仅监听 localhost、Token 校验、请求大小限制 |
| 无头浏览器被检测 | 🟢 低 | Playwright stealth 模式、User-Agent 轮换 |

### A. 术语表

| 术语 | 定义 |
|------|------|
| **Plugin** | 遵循 Plugin Registry 规范的动态模块，可在 AI 思考或工具执行阶段注入能力 |
| **CDP** | Chrome DevTools Protocol，通过 WebSocket 控制已有 Chrome 实例的协议 |
| **Playwright** | 微软开源的浏览器自动化库，支持 Chromium/Firefox/WebKit |
| **Snapshot** | AI 可见的 DOM 简化表示，包含可交互元素 ID 列表 |
| **Profile** | 浏览器登录态的快照（localStorage + Cookie），用于恢复会话 |
| **Reasoning Stream** | AI 深度思考过程中的文本流，由 LLM 服务端生成 |
| **SSRF** | Server-Side Request Forgery（服务端请求伪造） |
| **JSONPath** | JSON 数据的查询表达式语言（类似 XPath 之于 XML） |
