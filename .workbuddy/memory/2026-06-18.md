## ReAct 引擎响应延迟修复 (09:16)

### 根因分析
用户反馈 CDF auto model 下输入内容后 AI 很长时间才反馈。定位到 6 个延迟源：

1. **ReActExecutor 构造函数每次请求创建 13 个对象**（含 LongTermMemory SQLite 连接），每条消息都重新打开 DB
2. **moderate 级别任务触发 Planner LLM 调用**——在 ReAct 主循环前多一次完整 LLM 调用（无文本流给用户）
3. **simple 路径有工具调用时做 2 次 LLM 调用**——首次推理 + 工具执行 + 二次"最终响应"生成，即使首次已有充足内容
4. **无即时 SSE 反馈**——首 token 到达前用户看不到任何响应（LLM 推理可能 10-30s）
5. **longTermMemory.search 无错误兜底**——SQLite 异常直接中断主流程
6. **ReactStrategy 每次创建新 Observer/Planner 实例**

### 修复内容

**文件 1: `server/engine/reactExecutor.ts`**
- 构造函数改为懒加载：13 个对象用 getter 按需创建，`_budgetConfig` 延迟传入
- `execute()` 入口立即推送 `react_phase` SSE 事件（"AI 正在分析您的请求..."），让用户在首 token 前看到反馈
- 重置逻辑改为 `?.reset()`（仅已创建的模块才重置，避免强制初始化）
- Phase 0 规划仅对 `complex` 级别触发 LLM 规划，`moderate` 直进主循环（省 1 次 LLM 调用）
- simple 路径首次 LLM 调用已产出 >20 字符内容时，跳过第二次"最终响应" LLM 调用
- longTermMemory.search 包裹 try-catch，检索失败不阻塞主流程

**文件 2: `server/engine/executionStrategy.ts`**
- ReactStrategy 新增 static sharedObserver/sharedPlanner，懒加载复用，避免每请求重建

## DeepSeek API 400 tool_calls 配对修复 (16:12)

### 问题
DeepSeek API 报 400: "An assistant message with 'tool_calls' must be followed by tool messages responding to each 'tool_call_id'"

### 根因
上下文截断/压缩时把 `assistant(tool_calls)` 和对应的 `tool` 消息拆散：
1. `contextTruncate.ts` 从后往前逐条保留，可能保留 assistant(tool_calls) 但丢弃后续 tool 消息
2. `contextCompress.ts` 分割点 compressStartIdx 可能落在 tool 消息上，retained 开头出现孤儿 tool 消息

### 修复（三层防御）

**1. `contextTruncate.ts` — 原子分组截断**
- 新增 `groupMessagesAtomically()`: 将 `assistant(tool_calls)` + 后续连续 `tool` 消息分为一个原子单元
- 截断循环以单元为单位保留或丢弃，不再拆分配对
- 截断后调用 `sanitizeToolMessages()` 最终清理

**2. `sanitizeToolMessages()` — 安全网函数（导出）**
- 遍历消息，跟踪 `pendingToolCallIds` 集合
- `assistant(tool_calls)` 后无对应 `tool` 消息 → 移除 `tool_calls` 字段
- `tool` 消息找不到前面对应的 `assistant(tool_calls)` → 丢弃该 tool 消息
- 连续多个 `assistant(tool_calls)` 中间无 tool 消息 → 只保留最后一个的 tool_calls

**3. `contextCompress.ts` — 分割点修正**
- `compressStartIdx` 确定后向前扫描跳过连续 `tool` 消息（parent 在压缩区）
- 最终返回前调用 `sanitizeToolMessages()` 清理

**4. `aiClient.ts` — 发送前最终安全网**
- `callAIModelStream` 中 `effectiveMessages = sanitizeToolMessages(rawMessages)`
- 兜底所有场景，即使前面逻辑有遗漏也不会发送非法消息

## DeepSeek 400 修复 v2 — sanitizeToolMessages 两遍扫描 (19:01)

### 根因
v1.5.120 的 `sanitizeToolMessages` 单遍算法有缺陷：当 assistant(tool_calls=[A,B]) 只有 A 被响应时，清理 B 的 tool_calls 会移除 assistant 的 tool_calls 字段，但已加入 result 的 tool(A) 变成孤儿（前面没有 tool_calls），引发 400: "Messages with role 'tool' must be a response to a preceding message with 'tool_calls'"

### 修复

- Pass 1: 扫描所有 assistant(tool_calls)，向前查找哪些 tool_call_id 有响应
- Pass 2: 构建结果 — 只保留有响应的 tool_calls（部分保留）；只保留有匹配 assistant 的 tool 消息；空内容+无 tool_calls 的 assistant 消息直接跳过

## v2.8.2: SQLite 同步写入优化 — addMessage INSERT+UPDATE

### 问题
`server/dao/chat.ts` 的 `addMessage` 每次调用执行 3-5 次同步 SQL：
1. `db.prepare(INSERT).run()` — 插入消息
2. `db.prepare(UPDATE).run()` — 更新 sessions 元数据 (updatedAt/lastActiveAt/sessionDate)
3. `db.prepare(SELECT).get()` — 查标题（仅 user 消息）
4. `db.prepare(SELECT).get()` — 查消息数（仅首条 user 消息）
5. `db.prepare(UPDATE).run()` — 更新标题（仅首条 user 消息）

每次 `db.prepare()` 重新解析 SQL + autocommit 单独 fsync，事件循环阻塞 3-5ms。

### 修复（3 项优化）

**1. 模块级 prepared statements 缓存**
- `stmtCache` 对象缓存 5 条预编译语句
- `getStmts(db)` 首次调用时 prepare，后续直接返回缓存
- 消除每次 `addMessage` 的 SQL 解析开销

**2. INSERT 同步 + 会话元数据延迟**
- `addMessage` 仅同步执行 INSERT（WAL 模式追加写入 ~0.1ms）
- 会话元数据 UPDATE + 标题生成通过 `scheduleSessionUpdate()` 延迟到 `setImmediate`
- SSE 响应不再被 UPDATE/SELECT 阻塞

**3. 批量合并 + 事务**
- `pendingUpdates` 数组收集同一 tick 内的多个更新
- `flushSessionUpdates()` 在 `setImmediate` 中批量执行
- 全部包裹在单个 `db.transaction()` 中 — 1 次 fsync 替代 N 次

### 文件: `server/dao/chat.ts`

- 新增 `stmtCache` + `getStmts()` 预编译缓存
- 新增 `PendingSessionUpdate` 接口 + `pendingUpdates` 队列
- 新增 `scheduleSessionUpdate()` + `flushSessionUpdates()` 延迟批量执行
- `addMessage` 重写：1 次 INSERT + 1 次 `scheduleSessionUpdate`
