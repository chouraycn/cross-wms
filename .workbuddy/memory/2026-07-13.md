## 第十轮：技能无法回写文件 —— 根因分析（架构师高见远）

### 根因一句话
cross-wms「技能→文件写回→UI 渲染」在**协议层 + 识别层双重断点**：
1. SSE 协议无文件附件事件类型（`server/sse/sseTypes.ts` 联合类型 135/172/439/457 无 `file`/`attachment`/`downloadUrl`；`server/routes/agentChat.ts:248` switch 仅 `default→debug`，新 `file` 事件会被吞）。
2. `generatedFiles`（UI 渲染数据源）填充被**双重写死只认 `file_generateFile`**：后端 `server/engine/runChatSession.ts:551` `.filter(tc=>tc.name==='file_generateFile')` + 前端 `src/hooks/useAgentChat.ts:768` `if(toolName==='file_generateFile' && toolResult)`。
3. 工具层不产出 URL：`file_writeFile`(`fileTools.ts:339`) 返 `{success,path,bytesWritten}` 无 downloadUrl；`exec_command`(:417) 只 stdout/stderr；原生 handler(`skillLoader.ts:166-240`) 返 stdout/fetch —— 三者都不登记附件。
→ 技能只要不走 `file_generateFile`（实际多走 exec/原生 handler），产物既无 downloadUrl、也不被识别 → `GeneratedFileArtifactCard` 收不到数据 = 用户感知的"技能无法回写"。
- **UI 组件本身健康**：`GeneratedFileArtifactCard` + `BotMessageContent.tsx:513-557` 数据驱动、来源无关，只差数据流入。

### 改造核心思路（详见 deliverables/openclaw-vs-cdf-skill-fileback-analysis-2026-07-13.md）
- T1(P0) 新增与工具无关的 `file` SSE 事件（`server/sse/sseTypes.ts`）+ `agentChat.ts` switch 透传。
- T2(P0) 统一"生成文件注册器" `server/engine/generatedFileAttachment.ts`（makeFileId/buildGeneratedFilePayload/extractGeneratedFileFromToolResult/extractFilesFromMarkerText/emitFileEvent）；工具结果返回后自动 emit。
- T3(P1) 接通技能/exec 路径：复用 openclaw `FILE:/MEDIA:` 标记约定（`openclaw/src/engine/acp/event-mapper.ts:67`），`exec_command` 输出 `FILE:<绝对路径>` 即被识别 emit；原生 handler 执行后 emit；SKILL.md 约定产物落 generated dir 或走 `file_generateFile`。
- T4(P0) 前端 `useAgentChat.ts` 新增 `case 'file'` 追加进当前 assistant 消息 `generatedFiles`（按 fileId 去重），复用已存在卡片渲染。
- T5(P1/P2) 任意目录文件寻址 `/api/file/fs`、session 删除清理 generated 目录、补 file 事件回归测试。
- 字段对齐前端 `GeneratedFile` 类型 `src/types/chat.ts:131-148`（fileName/fileSize/mimeType/downloadUrl/previewUrl/sessionId...），额外加 fileId/toolCallId/source/skillId。

### 待用户确认（决定 T3 重心）
