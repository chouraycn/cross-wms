## ChatContainer pendingPermissions O(n)→O(1) 优化 (00:21)

### 问题
`pendingPermissions` useMemo 在 `permissionKey` 变化时对 `session.messages` 全量扫描 O(n)。流式期间 last message 权限状态变化时触发全量扫描，浪费性能。

### 优化
与 `permissionKey` 的 O(1) 策略对称——流式期间使用 ref 缓存 + 仅检查 last message：
- **3 个 ref 缓存**：`cachedPendingPermsRef`（非流式全量扫描结果 Map）、`lastMsgIdRef`（last message ID 跟踪）、`lastMsgPendingReqIdRef`（last message 的 pending reqId 跟踪）
- **流式 O(1)**：
  - last message 切换（新消息开始流式）→ 仅添加新 message 的 pending perm 到缓存 Map
  - same last message + pending state 不变 → 返回缓存引用（`pendingRequestsArray` 跳过 re-compute）
  - same last message + pending state 变化 → 从缓存 Map 移除旧 reqId、添加新 reqId
- **非流式 O(n)**：全量扫描，更新 3 个 ref 缓存

### 边界情况处理
- reqId 变更（工具 1→工具 2）：移除旧 reqId + 添加新 reqId ✓
- 权限解决（pending→approved/denied）：移除 reqId ✓
- 新 permission_request 到达：添加 reqId ✓
- last message 切换：不清除旧条目（旧 message 的 perm 仍有效）✓

## thinking/元数据事件 rAF 队列节流 (00:41)

### 问题
rAF 队列已统一所有 SSE 事件，但 `text` 事件有 `pendingContent` 缓冲 + 自适应分块渲染（天然节流），而 thinking/元数据事件每帧都创建状态更新（60fps）。虽然 MarkdownRenderer 有 150ms 节流，但 60fps 的状态更新仍触发：
- 每帧创建新 session 对象 + messages 数组 + streamingMsg 对象
- 每帧触发 React reconciliation → BotMessageContent areEqual 34 字段比较
- 每帧 ThinkingBlock 重渲染（被 MarkdownRenderer 150ms 节流挡住，但组件本身仍 re-render）

### 修复
在 `flushRender` 中添加元数据渲染节流：
- `METADATA_THROTTLE_MS = 100`（10fps）
- `lastMetadataFlush` 时间戳
- **纯元数据渲染**（`pendingContent.length === 0 && !shouldReschedule`）时检查节流窗口：
  - 窗口内 → 保持 `dirty=true`，reschedule rAF（下一帧重试）

- **text 事件不受影响**：有 `pendingContent` 或 `shouldReschedule` 时始终立即渲染

## 红黄绿按钮丢失 + 白屏根因分析与修复 (18:44)

### 问题现象
- v1.5.133 后红黄绿按钮反复丢失
- v1.5.151 出现白屏（启动后空白）

### 白屏根因（用户精准分析）
**问题定位**：`pywebview_app.py` 第 116-120 行的标题栏配置代码

**三个具体问题**：
| 问题 | 代码 | 后果 |
|------|------|--------|
| 样式被覆盖 | `setStyleMask_(NSWindowStyleMaskFullSizeContentView)` | 窗口样式被重置，丢失了其他必要样式（NSWindowStyleMaskTitled 等） |
| 标题隐藏 | `setTitleVisibility_(NSWindowTitleHidden)` | 标题栏区域高度变为 0，前端 `--pw-top: 38px` 避让逻辑失效 |
| 标题栏透明 | `setTitlebarAppearsTransparent_(True)` | WKWebView 内容直接覆盖标题栏区域，红黄绿按钮被遮挡 |

**为什么会导致白屏**：
- `setStyleMask_()` 是**覆盖式设置**，不是追加
- 当传入仅有 `NSWindowStyleMaskFullSizeContentView` 时，窗口失去了 `NSWindowStyleMaskTitled` 属性
- macOS 认为这是一个无标题栏窗口，重新布局
- WKWebView 内容视图被强制拉伸到整个窗口
- 但前端 CSS 有 `paddingTop: var(--pw-top, 0px)` 和 `WindowDragBar height={38}`
- 窗口布局与前端样式冲突 → **白屏**

**为什么红黄绿也丢了**：
- `NSWindowStyleMaskFullSizeContentView` 单独使用时，系统标题栏被隐藏
- 虽然 `frameless=False` 在创建窗口时保留了按钮，但 `setStyleMask_()` 调用后样式被覆盖

### 修复方案（v1.5.152）

**核心修复**：`setStyleMask_()` 改为**先读取、再追加**

```python
# ❌ 错误做法（覆盖）
target_window.setStyleMask_(NSWindowStyleMaskFullSizeContentView)

# ✅ 正确做法（追加）
current_mask = target_window.styleMask()
new_mask = current_mask | NSWindowStyleMaskFullSizeContentView
target_window.setStyleMask_(new_mask)
```
