## 2. 统一状态管理层

### 问题
项目有 4 个独立的 store（skillStore、chainStore、pluginStore、warehouseCapabilityStore），全部采用自定义事件总线模式（`listeners` Set），而非真正的 Zustand。缺少统一状态层，存在跨组件 prop drilling。

### 解决方案
新建 `src/stores/appStore.ts`（581 行），基于现有事件总线模式升级为单一 Store 实例：

- **4 个 Domain Slice**: skills、chains、plugins、warehouse
- **React Hooks**: `useAppStore()`（订阅全量）、`useAppStoreSelector(selector)`（按需订阅，性能优化）
- **Selectors**: `selectAllSkills`、`selectSkillById`、`selectSkillsByTrigger`、`selectEnabledPlugins`、`selectWarehouseFullView`
- **向后兼容**: 所有原有 store API 重新导出，现有代码无需修改

## 6. CI/CD 管道修复

### 根因
`onnxruntime-node` 的 postinstall 脚本会下载平台特定的原生二进制文件。在 CI 的 `ubuntu-latest` 环境中，网络限制或 OS 差异可能导致原生绑定编译/下载失败，从而阻塞 `npm ci`。而所有 ONNX 测试都已使用 `vi.mock()` mock，根本不需要真实的原生绑定。

### 修复措施

| 变更文件 | 修复内容 |
|---------|---------|
| `.nvmrc` | 新建，锁定 Node.js 版本为 22 |
| `package.json` | 添加 `engines` 约束（node>=20, npm>=10）；`onnxruntime-node` 移至 `optionalDependencies` |
| `vitest.config.ts` | 添加 `deps.optimizer.exclude: ['onnxruntime-node']`，确保 vitest mock 正确生效 |
| `.github/workflows/ci.yml` | test job 添加 `ONNX_DISABLE_NATIVE_DOWNLOAD=1` |
| `.github/workflows/pr-quality-gate.yml` | test job 添加 `ONNX_DISABLE_NATIVE_DOWNLOAD=1` |
