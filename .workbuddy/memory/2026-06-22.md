## 模型加载卡顿真正根因修复 (commit 4fc2b575)

### 背景
v1.5.201 commit `548c1187` 尝试修复模型加载卡顿（isLoading 初始值、MCP 并行连接、chunk 拆分等），但用户反馈仍无效果。深入排查发现真正的瓶颈在 Keychain 同步调用。

### 根因
`server/modelsStore.ts` → `loadModelsConfig()` → `injectApiKeys()` → `keychainStore.ts` `loadApiKey()` 使用 `execSync('security find-generic-password ...')` 同步调用 macOS Keychain：
- 每个 execSync 阻塞 50-200ms
- N 个模型 = N × 50-200ms 事件循环阻塞
- `GET /api/models` 拿到 key 后**立即脱敏丢弃** — 纯浪费
- 缓存 TTL 仅 5 秒，频繁失效后重新阻塞

### 修复方案（3 项）

#### Fix 1: loadModelsConfig 增加 skipKeyInjection 参数
- `server/modelsStore.ts` 函数签名改为 `loadModelsConfig(options?: { skipKeyInjection?: boolean })`
- `skipKeyInjection: true` 时跳过 `injectApiKeys()`、自动禁用检查、`extractAndSaveApiKey`
- 热缓存时返回脱敏副本（不更新主缓存）
- 冷缓存时返回迁移后的数据（不注入 key、不更新主缓存）
- `CACHE_TTL_MS` 从 5000 改为 30000（30 秒）

#### Fix 2: GET /api/models 使用 skipKeyInjection
- `server/routes/models.ts` GET `/` 路由调用 `loadModelsConfig({ skipKeyInjection: true })`
- 保留脱敏作为安全兜底

#### Fix 3: 启动时预热缓存
- `server/index.ts` `server.listen()` 回调中 `loadModelsConfig().catch(...)` 预热
- 不 await，不阻塞启动流程

### 同一 commit 中的其他修复
- **ThinkingBlock 卡在"思考中"**: 新增 `thinkingDone` 标志，收到首个 text 事件时标记 thinking 结束
- **Electron SSE 渲染不刷新**: `isDesktopApp()` 增加 Electron 检测，降级为 `setTimeout(fn,16)` 避免 `backgroundThrottling` 节流 rAF
- **keep_alive 格式**: 从 SSE 注释格式改为 JSON 格式
- **Phase 1 Agent 可视化**: ExecutionTrace + AgentStatusIndicator 组件集成

### 验证
- TypeScript: `tsc --noEmit` 0 errors ✅
- 测试: `vitest run server/__tests__/modelsStore.test.ts` 6/6 passed ✅
- 影响范围: AI 推理路径（chainExecutor/chatService/memoryExtractor）不受影响，仍使用完整 `loadModelsConfig()`

## scheduleFrame 动态检测修复 (commit d3724a0d)

### 现象
AI 输入后卡在"思考中"不动，点击页面后内容突然全部出现。

### 根因（深层时机 bug）
`src/hooks/useChat.ts`：`const IS_PYWEBVIEW = isDesktopApp()` 是**模块级常量**，在 JS bundle 加载时（比 window.onload 早）求值。此时 pywebview 尚未注入 `window.pywebview`，`isDesktopApp()` 返回 false，`scheduleFrame` 被固定为 `requestAnimationFrame`。WKWebView 空闲时暂停 rAF → SSE 数据积压 → 点击才刷新。

### 修复 (v2.9.0)
移除 `IS_PYWEBVIEW` 常量，`scheduleFrame` 改为每次调用时动态调用 `isDesktopApp()`，确保 pywebview 注入后首次渲染即正确使用 `setTimeout(fn, 16)`。

- 移除 `import { isDesktopApp }` 未使用导入
- keep_alive 间隔 15s → 5s（用户更快感知连接活跃）

### 根因链总结
1. v2.9.0: `IS_PYWEBVIEW` 模块级常量在 pywebview 注入前求值 → 永远 false → 用 rAF
2. v2.9.0-fix (v1.5.204): 改为动态检查，但首次调用仍可能返回 false → rAF 被调度 → WKWebView 暂停 → renderHandle 卡死

## v1.5.206: 主路径 keepAliveTimer 缺失修复 (commit 02a55b52)

### 问题
用户反馈 auto model 模式下卡在"思考中"。v1.5.205 的 rAF 修复解决的是渲染层问题，
本次是**后端 SSE 心跳缺失**导致的另一层问题。

### 根因
`chatService.ts` 有两条代码路径：
1. `executeFromQueue` 函数（队列模式）— line 248 启动了 `keepAliveTimer`（5s 间隔）
2. `handleChat` 主函数（非队列模式/默认）— line 718 声明了 `keepAliveTimer = null` 但**从未调用 setInterval**

auto 模式走的是主路径（非队列），导致：
- 后端发 `init` 事件后，做上下文压缩 + ONNX 记忆搜索（5-10s）
- 调 AI API，推理模型首 token 可达 20-30s
- 期间无 keep_alive 事件 → 前端 30s 心跳超时 → "SSE 连接超时" 错误
- 用户盯着"思考中..."旋转圈 30 秒后看到错误

### 修复（3 项）
1. **chatService.ts**: 主路径 init 事件后启动 `keepAliveTimer`（5s 间隔，与 executeFromQueue 一致）
2. **useChat.ts**: SSE 心跳超时 30s → 60s（安全兜底，推理模型首 token 可达 30-40s）
3. **useChat.ts**: 错误 catch 路径设置 `thinkingDone = true`（防止 ThinkingBlock 卡住）

## v1.5.207: tool_calls 400 错误 + 偶尔卡在思考中（终版修复）(commit 146ee4de)

### 问题
1. **API 400 "insufficient tool messages following tool_calls message"** — ReAct 执行器 P1-2 输出校验注入的 system 消息打断了 tool_calls 序列
2. **偶尔仍卡在思考中** — 外层 catch 不发 done 事件，前端永远等不到 SSE 结束

### 修复 1: tool_calls 400 错误（三层防御）

#### 源头修复 — `reactExecutor.ts`
- **根因**: P1-2 输出校验在 `assistant(tool_calls)` 和 `tool` 消息之间注入 `system` 消息，打断了 OpenAI 要求的 `assistant(tool_calls) → tool` 配对序列
- **修复**: 用 `pendingSystemMessages[]` 收集 system 提示，先推送所有 tool 消息（紧跟 assistant(tool_calls)），最后再注入 system 提示

#### 中间安全网 — `contextTruncate.ts`
- **新增 Pass 3.5 重排序逻辑**: 扫描 `validated[]` 数组，如果 `assistant(tool_calls)` 和对应 `tool` 消息之间被 `system`/`user` 消息插入，自动将 tool 消息移到 assistant(tool_calls) 之后
- 算法: 遇到 `hasToolCalls(msg)` 时，向后扫描收集匹配的 tool 消息和剩余消息，先 push tool 消息再 push 剩余消息

#### 最终安全网 — `aiClient.ts`
- `callAIModelStream` catch 块新增 400 tool_calls 错误重试: strip 所有 tool_calls/tool 消息，将 `assistant(tool_calls)` 降级为普通文本后重试（tools 参数传 undefined）

### 修复 2: 偶尔卡在思考中

#### 外层 catch 不发 done 事件 — `chatService.ts`
- **根因**: `handleChat` 外层 catch（line 1435）在 `res.headersSent` 时只发 `{ type: 'error' }` 不发 `{ type: 'done' }`，若 `res.write()` 抛异常则前端永远收不到结束信号
- **修复**: 始终发送 `error + done` 事件（带 errorCode/errorMessage），用 try-catch 包裹 res.write() 防止二次抛出

#### keepAliveTimer 泄漏 — `chatService.ts`
- **根因**: `keepAliveTimer` 在 try 块内声明，catch 块访问不到，若 error 发生在 innermost finally 之前则定时器泄漏
- **修复**: 将 `keepAliveTimer` 声明移到 try 外（两个函数都改），外层 catch 和 executeFromQueue catch 块中添加 `clearInterval(keepAliveTimer)` 清理

### UI 重构（同 commit）
- 提取 `SettingsDialogShell` 共享组件，统一 `AISettingsDialog` 和 `ToolManagementDialog` 的对话框外壳布局
